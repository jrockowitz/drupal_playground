Study detail page — data structure approach

How should the nested JSON be rendered on the study detail page?

A

Flat table — dotted key paths

One table, two columns. Every leaf value gets a row with its full dotted path as the key.

Study: NCT04001699
Field path Value
protocolSection.identificationModule.nctIdNCT04001699
protocolSection.identificationModule.briefTitleA Study of ABC123 in Patients With...
protocolSection.statusModule.overallStatusRECRUITING
protocolSection.statusModule.startDateStruct.date2021-03-15
protocolSection.sponsorCollaboratorsModule.leadSponsor.namePfizer Inc.
......

Pros

  • Dead simple to implement
  • Shows exact API paths
  • Easiest to scan for specific field

Cons

  • Long key paths are verbose
  • No visual grouping
B

Nested <details> — mirrors JSON structure

Top-level sections are collapsible <details> elements. Leaf values are rendered with item elements inside.

Study: NCT04001699
protocolSection
identificationModule
nctId: NCT04001699
briefTitle: A Study of ABC123 in Patients With...
statusModule
...
derivedSection
...

Pros

  • Matches actual API structure
  • Collapsible — hides noise
  • Short leaf-level key names

Cons

  • Many click-throughs to find a field
  • More complex to build
C

One <details> per top-level module — flat table inside

Each protocolSection.*Module becomes a collapsible <details>. Inside each one, a flat two-column table shows the leaf key (short name) and value.

Study: NCT04001699
identificationModule
nctIdNCT04001699
briefTitleA Study of ABC123 in Patients With...
statusModule
overallStatusRECRUITING
startDateStruct.date2021-03-15
sponsorCollaboratorsModule
...

Pros

  • Grouped and scannable
  • Shorter key names in table
  • One click to expand a section

Cons

  • Slightly more complex than A
  • Arrays of objects need extra handling
