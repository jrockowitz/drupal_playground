# Term Reference Implementation Notes

## Goal

Term Reference is a contrib-ready Drupal module that adds a `References` tab to
taxonomy term pages and lets editors add or remove the current term from
eligible taxonomy term entity reference fields.

## Implemented Architecture

Term Reference discovers configurable entity reference fields that target
taxonomy terms and keys them by `{entity_type_id}.{field_name}`.

The module uses one route:

```text
/taxonomy/term/{taxonomy_term}/references/{field}
```

The `field` parameter is optional. Without a field, the form uses the accessible
field count to choose the page behavior:

- One accessible field: render the add/remove form immediately.
- Multiple accessible fields: render an `admin_block_content` chooser.
- No accessible fields: deny route access.

With a field, the form renders the add/remove form for that field. The field
parameter contains values like `node.field_tags` or `media.field_tags`.

The page title is generated by `TermReferenceForm::title()` and renders as:

```text
Add references to %term
```

## Discovery

`term_reference.discovery` scans all fieldable content entity types and their
bundle field definitions. It includes configurable `entity_reference` fields
whose `target_type` is `taxonomy_term`. For a specific taxonomy vocabulary, a
field is eligible when its handler settings either allow all vocabularies or
include the vocabulary ID in `target_bundles`.

Discovery returns arrays keyed by `{entity_type_id}.{field_name}` with:

- `id`.
- `entity_type_id`.
- `entity_type_label`.
- `bundle_entity_type_label`.
- `field_name`.
- `field_label`.
- `vocabulary_id`.
- `bundles`.

The discovery service caches results in `cache.discovery` with the
`term_reference:fields` cache tag. `TermReferenceHooks::fieldConfigClearCache()`
invalidates that cache tag whenever field config is inserted, updated, or
deleted. Passing an account to `getFieldsForVocabulary()` filters results to
fields on bundles the account can update and field definitions the account can
edit.

## Form Flow

`TermReferenceForm` receives the taxonomy term entity and optional field route
parameter. The taxonomy term route parameter is converted to a term entity by
Drupal routing.

When the field parameter is present, the form splits it into entity type ID and
field name, asks discovery for the matching field for the current term
vocabulary, and builds the management form.

When the field parameter is absent, the form asks discovery for fields editable
by the current account. One accessible field is used immediately. Multiple
accessible fields render a chooser. Chooser links use the same
`term_reference.references` route with the selected field parameter.

The add fieldset uses Drupal core's `entity_autocomplete` element:

- Element name: `entities`.
- `#target_type`: discovered entity type ID.
- `#tags`: `TRUE` for multiple selections.
- `#selection_handler`: `default`.
- `#selection_settings['target_bundles']`: discovered eligible bundles.
- `#validate_reference`: `TRUE`.

Form descriptions use `bundle_entity_type_label` from discovery so target
bundles are introduced with entity-specific labels such as `Content types` and
`Media types`.

The add form uses Drupal core autocomplete validation. Drupal validates
autocomplete selections first, then `TermReferenceForm::validateAddReference()`
checks that every selected entity can be managed.

The existing references fieldset contains a render-array table with Label, ID,
Bundle, Published, and Operations columns. Remove checkboxes are only rendered
for entities the current account can manage.

Add and remove support Drupal AJAX and normal form submissions. The AJAX
response replaces the `term-reference-form-wrapper`, refreshes status messages,
announces state changes, and moves focus to the rebuilt autocomplete field or
existing references fieldset.

## Access Model

The module uses Drupal entity and field access without a custom permission.

`TermReferenceForm::access()` checks the route. With no field parameter, access
is allowed when the current account can manage at least one discovered field for
the term vocabulary. With a field parameter, access is allowed only when that
field exists for the vocabulary and `TermReferenceManager::accessReference()`
allows it.

`TermReferenceManager::accessReference()` requires:

- Taxonomy term update access.
- Field edit access for at least one eligible field instance, checked through
  the target entity type access handler with `fieldAccess('edit', ...)`.

Add and remove operations also require:

- Target entity update access.
- The configured field to exist on the target entity.
- The target entity bundle to be one of the eligible bundles for the field.
- Field edit access on the target entity field item list.

## Manager Behavior

`term_reference.manager` provides the content entity operations used by the
form:

- `loadReferencingEntities()` queries entities whose configured field contains
  the current term, filters by eligible bundles, access-checks the query, sorts
  by entity label when available, and returns content entities.
- `addReference()` appends the term only when the field does not already contain
  it.
- `removeReference()` removes only the current term from the configured field
  and leaves other referenced terms intact.

## Test Coverage

The functional test creates:

- A `tags` vocabulary.
- `page` and `article` node types.
- An `image` media type.
- `field_tags` on `node.page`, `node.article`, and `media.image`.
- A `topics` vocabulary with one eligible `node.field_topics` field.

It verifies:

- Discovery returns `node.field_tags` and `media.field_tags`.
- The taxonomy term page renders the `References` primary task.
- The primary route renders chooser links when multiple fields are available.
- The primary route renders the management form directly when one field is
  available.
- Field-specific URLs still use the same `term_reference.references` route with
  the optional field parameter.
- The content field can add Basic page and Article nodes.
- The media field can add Image media.
- The table shows label, ID, bundle, published state, and operations.
- Remove clears only the selected term reference.
- A limited user receives a 403 response.

The FunctionalJavascript test verifies AJAX add/remove behavior on a direct
field URL.

Kernel tests verify the discovery, manager, and hook interfaces and confirm
field config changes clear discovery caches. Shared entity schema, config, and
fixture helpers live in `TermReferenceKernelBase`.
