AI Schema.org JSON-LD (ai_schemadotorg_jsonld) adds AI-assisted Schema.org JSON-LD generation to supported Drupal content entity bundles by connecting AI Automators, Field Widget Actions, and JSON Field. It stores the generated JSON-LD in a field and injects saved values into canonical entity pages as <script type="application/ld+json"> tags.

Features

A glue module for generating, editing, storing, and outputting Schema.org JSON-LD from Drupal entity content.
  • Adds a generated field_schemadotorg_jsonld field to supported, fieldable content entity bundles with canonical routes.
  • Creates a matching AI Automator configuration that uses the bundle's entity-type prompt and the Field Widget Actions "Generate Schema.org JSON-LD" button.
  • Uses JSON Field for storage and display, with optional json_field_widget support for a richer JSON editor widget.
  • Ships with node support configured out of the box and can also support additional canonical, fieldable content entity types such as media, taxonomy terms, block content, comments, and users.
  • Uses the same builder behavior from the settings form, the Drush command, and the recipe config action so generated field, automator, and display configuration stay consistent.

Requirements and installation

AI Schema.org JSON-LD requires Drupal core 11.3 or later, AI with the ai_automators submodule, Field Widget Actions, and JSON Field. The optional json_field_widget submodule enables the json_editor widget for the generated field and for the admin default_jsonld setting.

  • Required modules: ai:ai_automators, field_widget_actions:field_widget_actions, and json_field:json_field.
  • Optional module: json_field:json_field_widget.
  • Development dependency: Token improves token browsing in the settings form, but the module's runtime requirements come from the required modules above.

After enabling ai_schemadotorg_jsonld, configure an AI provider at /admin/config/ai/settings, then open /admin/config/ai/schemadotorg-jsonld to enable entity types and select bundles.

Configuration

The settings form at /admin/config/ai/schemadotorg-jsonld is the main admin workflow. It shows one section per enabled entity type and an "Enabled entity types" section for turning on additional supported entity types.

  • By default, node is enabled and ships with a curated default prompt.
  • Saving newly enabled entity types adds default prompt and default_jsonld settings for those entity types.
  • Selecting bundles and saving builds the generated field, AI automator, AI automator status support, form display, and view display configuration for each selected bundle.
  • Each entity type section lets you manage bundle selection, a token-based default prompt, and optional static default_jsonld that is injected on canonical pages for bundles that already have the generated field.
  • Saved entity edit forms can expose an "Edit prompt" modal for site configuration administrators when the development setting is enabled.

Only supported content entities that are fieldable and have canonical routes can use this module. Existing field storage is respected, so enabled entity-type settings are not removed while generated Schema.org JSON-LD storage still exists.

Drush and recipes

The module includes a Drush command and a recipe config action plugin. Both delegate to the same builder service used by the settings form.

Add the field to a specific bundle:

drush ai_schemadotorg_jsonld:add-field node page

Add the field to all current bundles for an entity type, or to a bundle-less entity type such as user:

drush ai_schemadotorg_jsonld:add-field node '*' drush ai_schemadotorg_jsonld:add-field user

Use the addField config action in a recipe to apply the same generated field setup during installation or site assembly:

config: actions: ai_schemadotorg_jsonld.settings: addField: entity_type: node bundles: ['article', 'page']

To target all bundles that exist when the recipe runs, use a wildcard bundle list:

config: actions: ai_schemadotorg_jsonld.settings: addField: entity_type: node bundles: ['*']

The repository also includes the Drupal Playground AI Schema.org JSON-LD recipe, which installs the module, enables related submodules, and applies addField to all current node bundles.

Prompts, tokens, and output

AI Schema.org JSON-LD ships prompt files in prompts/entity_types/ for common entity types including node, media, taxonomy_term, block_content, comment, and user. When you enable a new supported entity type, the module uses a matching prompt file when one exists and otherwise builds a minimal default prompt from the entity type definition.

  • Prompts are token-based and can include the rendered entity content token such as [node:ai_schemadotorg_jsonld:content].
  • The content token renders the entity as the anonymous user in the site default theme, converts root-relative URLs to absolute URLs, and strips outer wrapper markup to keep prompt input cleaner for the LLM.
  • The generated field stores the JSON-LD returned by the model, and the module outputs compact JSON in application/ld+json script tags on canonical entity pages.
  • Depending on configuration, a page can emit saved entity JSON-LD, entity-type-level default_jsonld, and breadcrumb JSON-LD from the optional breadcrumb submodule.

New entities must be saved before Schema.org JSON-LD can be generated, because the field widget action works on saved entity edit forms.

Submodules

AI Schema.org JSON-LD Log

ai_schemadotorg_jsonld_log adds prompt and response logging tools for AI Schema.org JSON-LD. It provides a log UI at /admin/config/ai/schemadotorg-jsonld/log, a "View log" link from supported entity edit forms, CSV export, filtered per-entity log views, and automatic cleanup when logged entities are deleted.

AI Schema.org JSON-LD Breadcrumb

ai_schemadotorg_jsonld_breadcrumb attaches a BreadcrumbList JSON-LD block on canonical pages when breadcrumb data is available.

Getting help

Use the Drupal.org issue queue to report bugs, request improvements, and discuss follow-up work. When reporting a problem, include the Drupal version, enabled AI provider, entity type and bundle, and whether the issue happens in the settings form, the field widget action, the Drush command, or recipe-based installation.
